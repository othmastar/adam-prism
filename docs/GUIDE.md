## 1. نظرة عامة على المشروع

### ما هو آدم بريزم؟
آدم بريزم هو **توأم رقمي شخصي** يتطور ليصبح **نظام إدارة مؤسسي بالذكاء الاصطناعي** يساعد القيادات في:
- إدارة الشركات واتخاذ القرارات
- التنبؤ بالمخاطر والفرص
- كشف الفساد والانحرافات
- التعلم المستمر من تفاعلاتك

### لماذا "بريزم"؟
المنشور (Prism) يحوّل الضوء الأبيض إلى ألوان الطيف. كذلك آدم يحوّل بياناتك الخام إلى رؤى وأفكار ملونة.

### المبادئ الأساسية
- **الأولوية**: الجودة > الدقة > السرعة — بدون تنازلات
- **العمل المحلي**: كل شئ يعمل على جهازك، لا سحابة
- **التعلم المستمر**: آدم يتعلم منك ويحفظ نمط تفكيرك
- **الأخلاق أولاً**: كل رد يمر عبر بوابة أخلاقية قبل الإرسال

## 2. المتطلبات والتثبيت

### المتطلبات الأساسية

| العنصر | الحد الأدنى | الموصى به |
|--------|-------------|-----------|
| نظام التشغيل | Ubuntu 20.04+ | Ubuntu 22.04 LTS |
| الرام | 16 GB | 32 GB |
| كرت الشاشة | NVIDIA RTX 3060 (12GB) | NVIDIA RTX 4090 (24GB) |
| التخزين | 50 GB حر | 200 GB SSD |
| الإنترنت | للتثبيت فقط | سريع |

### الخطوة 1: تثبيت Ollama (مشغّل GGUF)

```bash
# تثبيت Ollama
curl -fsSL https://ollama.com/install.sh | sh

# تحميل النموذج (Gemma 3 4B - حوالي 3GB)
ollama pull gemma3:4b

# تحميل نموذج التضمين (Nomic Embed - حوالي 274MB)
ollama pull nomic-embed-text

# اختبار التشغيل
ollama run gemma3:4b "مرحباً، من أنت؟"
```

**ما هو Ollama؟**
Ollama هو مشغّل نماذج ذكاء اصطناعي محلي. يقرأ ملفات GGUF (النسخة المضغوطة من النماذج) ويشغلها على كرت الشاشة. لا يحتاج إنترنت بعد التحميل.

**ما هو GGUF؟**
GGUF (GPT-Generated Unified Format) هو صيغة ضغط خاصة بنماذج الذكاء الاصطناعي. الميزة:
- حجم أصغر (النموذج الأصلي 8GB → GGUF 3GB)
- يعمل على أجهزة أقل مواصفات
- سرعة استدلال أعلى
- متوافق مع llama.cpp و Ollama

### الخطوة 2: تثبيت Qdrant (قاعدة البيانات الشعاعية)

```bash
# باستخدام Docker (الأسهل)
docker pull qdrant/qdrant:latest
docker run -d \
    --name qdrant \
    -p 6333:6333 \
    -p 6334:6334 \
    -v ./qdrant_data:/qdrant/storage \
    qdrant/qdrant:latest

# أو بدون Docker
# حمّل من: https://github.com/qdrant/qdrant/releases
```

**ما هو Qdrant؟**
Qdrant هو قاعدة بيانات شعاعية (Vector Database). يخزن المعرفة كأرقام (embeddings) ويبحث فيها بالمعنى لا بالكلمات. لما تسأل عن "مبدأ عدم اليقين"، يربط تلقائياً بفيزياء الكم وفلسفة العلم وإدارة المخاطر — لأن المعاني متقاربة.

## 4. كل موديول بالتفصيل

### 4.1 المحرك الرئيسي (core/engine.py)

**الوظيفة**: المحرك المركزي الذي يربط كل الموديولات.

**كيف يعمل**:
```
رسالة المستخدم
    ↓
[1. فحص الأمان] → SecurityCoordinator
    ↓
[2. تصنيف القصد] → تحديد الوضع المعرفي (7 أوضاع)
    ↓
[3. بناء السياق] → من الذاكرة + المعرفة
    ↓
[4. البحث في Qdrant] → معرفة ذات صلة
    ↓
[5. التوليد عبر Ollama] → GGUF model
    ↓
[6. التقييم الأخلاقي] → EthicsGate
    ↓
[7. التسجيل في النوته] → AdamNotebook
    ↓
[8. حفظ في الذاكرة] → Qdrant
    ↓
الرد النهائي
```

**الأوضاع المعرفية السبعة**:
- `analyst` (محلل): تحليل بيانات وأنماط
- `builder` (بنّاء): بناء وتنفيذ حلول
- `corrector` (مصحح): مراجعة وتصحيح
- `engineer` (مهندس): تصميم أنظمة
- `researcher` (باحث): بحث واستكشاف
- `communicator` (متواصل): شرح وتوضيح
- `strategist` (استراتيجي): تخطيط واستراتيجية

**إعدادات مهمة**:
- `context_window: 8192` - حجم السياق (tokens)
- `token_budget: 8100` - الميزانية المتاحة
- `model_name: "gemma3:4b"` - اسم النموذج في Ollama

---

### 4.2 نظام الذاكرة (memory/memory_system.py)

**الوظيفة**: ذاكرة طويلة المدى + قصيرة المدى مع Qdrant + Nomic Embedding.

**المجموعات الخمس في Qdrant**:

| المجموعة | المحتوى | مثال |
|----------|---------|-------|
| knowledge | معرفة عامة | "مبدأ عدم اليقين ينص على أنه لا يمكن قياس موقع وسرعة الجسيم معاً بدقة" |
| conversations | محادثات | سؤال: "اشرح ميكانيكا الكم" → جواب: "..." |
| patterns | أنماط تفكير | "المستخدم يفضل الإجابات المنهجية" |
| summaries | ملخصات | ملخص كتاب "تفكير سريع وبطيء" |
| connections | روابط | "مبدأ عدم اليقين ↔ اتخاذ القرار في ظل الغموض" |

**كيف يعمل التضمين (Embedding)**:
```
نص: "مبدأ عدم اليقين"
    ↓
Nomic Embed Model (محلي عبر Ollama)
    ↓
[0.023, -0.156, 0.087, ...] ← 768 رقم
    ↓
يُخزن في Qdrant
    ↓
لما تسأل: "ما هو مبدأ هايزنبرغ؟"
    ↓
سؤالك → أرقام [0.025, -0.148, 0.091, ...]
    ↓
Qdrant يقارن ويجد: المعنى متطابق! (رغم اختلاف الكلمات)
```

### 4.3 البوابة الأخلاقية (ethics/ethics_gate.py)

**الوظيفة**: تقييم كل رد حسب القوانين الأربعة قبل الإرسال.

**القوانين بالأولوية**:
1. **العدالة (40%)**: هل الرد منصف وصادق وغير متحيز؟
2. **التعلم (30%)**: هل يساعد المستخدم على النمو؟
3. **البقاء (20%)**: هل يحافظ على الأمان؟
4. **الإبداع (10%)**: هل مبتكر ويحل المشكلة؟

**آلية العمل**:
```
الرد المُولَّد
    ↓
[1. فحص المحرمات المطلقة] → إيذاء/انتهاك خصوصية/تزوير
    ↓ محظور → رفض فوري
    ↓
[2. تقييم بالأوزان عبر النموذج] → كل قانون من 0 إلى 1
    ↓
[3. حساب المجموع المرجح] → العدالة×0.4 + التعلم×0.3 + البقاء×0.2 + الإبداع×0.1
    ↓
[4. لو النتيجة < 0.5] → رفض أو تعديل
    ↓
الرد النهائي
```

### 4.6 التلخيص الحي الهرمي (pipeline/live_summarizer.py)

**المشكلة التي يحلها**:
ملف 500 صفحة = ~200,000 tokens. السياق المتاح = ~8,000 tokens. مستحيل تقرأه دفعة واحدة.

**الحل**:
```
الملف (500 صفحة)
    ↓ يُقسّم لأجزاء
الجزء 1 → يُلخص → ملف حي: summary_01.md
الجزء 2 + سياق الجزء 1 → يُلخص → summary_02.md
الجزء 3 + سياق الجزء 2 → يُلخص → summary_03.md
    ...
كل الملخصات → ملخص شامل + خريطة مفاهيم + بيانات تدريبية
```

**المخرجات**:
1. `master_summary.md` — ملخص شامل نهائي
2. `concepts.json` — خريطة المفاهيم مع الروابط
3. `training.jsonl` — بيانات تدريبية (سؤال-جواب) جاهزة
4. `live_summary.md` — ملف حي يتحدث في الوقت الفعلي

### المرحلة 2: التدريب الأولي (الأسبوع 4-5)

**الهدف**: ضبط النموذج ليتعلم أسلوبك

**طريقة التدريب مع GGUF**:

هناك طريقتان رئيسيتان:

#### الطريقة A: LoRA على Transformers ثم تحويل لـ GGUF (الموصى بها)

```
الخطوة 1: تحويل GGUF → النموذج الأصلي (إذا لزم)
الخطوة 2: تدريب LoRA على النموذج الأصلي
         (يستخدم فقط جزء صغير من الأوزان - ~1-5%)
الخطوة 3: دمج LoRA مع النموذج الأساسي
الخطوة 4: تحويل النتيجة لـ GGUF
الخطوة 5: استخدام GGUF الجديد مع Ollama

التفاصيل:
──────────────────────────────────────
1. تثبيت أدوات التدريب:
   pip install torch transformers peft datasets

2. إعداد LoRA:
   - rank (r): 8-16 (أصغر = أقل ذاكرة)
   - target_modules: ["q_proj", "v_proj", "k_proj", "o_proj"]
   - learning_rate: 2e-4
   - epochs: 3-5
   - batch_size: 4 (مع gradient accumulation)

3. التدريب:
   python scripts/train_lora.py \
     --model google/gemma-2-4b \
     --data training_data.jsonl \
     --output ./lora_output \
     --epochs 3

4. الدمج والتحويل:
   python scripts/merge_and_convert.py \
     --base google/gemma-2-4b \
     --lora ./lora_output \
     --output ./adam-prism-gguf

5. التحميل في Ollama:
   ollama create adam-prism -f Modelfile
   # Modelfile content:
   # FROM ./adam-prism-gguf.gguf
   # PARAMETER temperature 0.7
   # PARAMETER num_ctx 8192

6. التحديث في الإعدادات:
   config/default.json → model_name: "adam-prism"
```

#### الطريقة B: تدريب مباشر على llama.cpp

```
llama.cpp فيه نظام تدريب مباشر لكنه:
- أقل مرونة من LoRA
- أصعب في الإعداد
- لا يدعم كل أنواع LoRA

لذلك نوصي بالطريقة A
```

**الاحتياطات ضد النسيان الكارثي (Catastrophic Forgetting)**:
```
المشكلة: لو درّبت النموذج على بياناتك فقط، ينسى معلوماته العامة

الحل: Replay Buffer
1. اختر 1000 سؤال-جواب متنوع من المعرفة العامة
2. أضفها لبياناتك التدريبية (20% عام + 80% خاص)
3. هذا يضمن: يتعلم أسلوبك + يحتفظ بمعرفته
```

**معايير النجاح**: آدم يرد بأسلوبك ويحتفظ بمعرفته العامة.

**الروتين اليومي**:
```

```

**آلية التعلم المستمر**:

```
3. أنماطك تُستخرج وتُخزّن
4. كل شهر: تُضاف البيانات الجديدة لبيانات تدريبية

```

## 6. كيف يعمل النظام خطوة بخطوة

### مثال كامل: محادثة عادية

```
أنت: "آدم، لخص لي آخر اجتماع"

الخطوة 1: الأمن
→ SecurityCoordinator: فحص المصادقة ✓
→ فحص السلوك: لا أنماط مشبوهة ✓
→ لا يحتاج تأكيد ✓

الخطوة 2: تصنيف القصد
→ النموذج يحلل: "لخص" = تلخيص + "اجتماع" = عمل
→ الوضع: analyst (محلل)
→ القصد: summarization
→ الثقة: 0.9

الخطوة 3: بناء السياق
→ آخر 10 رسائل من المحادثة
→ ذكريات ذات صلة من Qdrant
→ معرفة عن الاجتماعات السابقة

الخطوة 4: البحث في Qdrant
→ "آخر اجتماع" → embedding → بحث
→ يجد: ملاحظات اجتماع الأسبوع الماضي
→ 3 نتائج ذات صلة

الخطوة 5: التوليد
→ Ollama يولّد الرد بناءً على:
  - السياق من المحادثة
  - المعرفة من Qdrant
  - تعليمات النظام (الوضع analyst)

الخطوة 6: التقييم الأخلاقي
→ العدالة: 0.8 (منصف)
→ التعلم: 0.7 (يفيد المستخدم)
→ البقاء: 0.9 (آمن)
→ الإبداع: 0.6 (كفاية)
→ المجموع: 0.8×0.4 + 0.7×0.3 + 0.9×0.2 + 0.6×0.1 = 0.77 ✓

الخطوة 7: التسجيل في النوته
→ يكتب في daily/2025-01-15.md:
  "[14:30] دورة #42 | وضع: analyst | سؤال عن تلخيص اجتماع"

الخطوة 8: حفظ في Qdrant
→ المحادثة تُخزّن بالـ embedding
→ المرة الجاية لما تسأل عن الاجتماع، يلقاه

الخطوة 9: الرد
→ "الاجتماع تناول 3 محاور: ..."
→ المدة: 2.3 ثانية
```
